# Стандарты документирования

> Doc-ID: DOC-STANDARDS-1 | Дата: 2026-06-02 | Связанные: [ARCH-v2], [CHANGELOG]

## 1. Doc-ID система

Каждый `.md` файл в `docs/` ОБЯЗАН иметь заголовок:
```markdown
> Doc-ID: <УНИКАЛЬНЫЙ-КОД> | Дата: <ДАТА> | Связанные: [DOC-ID-1], [DOC-ID-2]
```

**Правила:**
- Doc-ID уникален навсегда. При обновлении файла дата меняется, Doc-ID — нет.
- Префиксы: `ARCH-` (архитектура), `FEAT-` (feature), `GUIDE-` (гайд), `REF-` (reference), `BUG-` (баг), `CHANGELOG`, `DOC-` (документация)
- При удалении файла Doc-ID помечается как `[DEPRECATED]` в CHANGELOG

## 2. Cross-Reference

- Ссылка на другой документ: `см. [DOC-ID]` или `подробнее: [DOC-ID](filename.md)`
- В заголовке файла указывай `Связанные: [ID-1], [ID-2]` — список всех связанных Doc-ID
- При изменении файла проверь, не сломались ли ссылки из других документов

## 3. Detailed Response Format (ОБЯЗАТЕЛЬНО)

После КАЖДОГО изменения кода, анализа, или создания нового функционала пиши ответ в формате из 7 обязательных секций:

### 3.1 [Контекст]
- Какая проблема решается
- Как воспроизвести (шаги)
- Какие симптомы (ошибки, логи, скриншоты)
- Версия/окружение (OC, версия opencode, версия Python)

### 3.2 [Исследование]
- Какие файлы прочитаны (с номерами строк)
- Какие SQL-запросы выполнены и их результаты
- Какие команды запущены и их вывод
- Какие альтернативные источники проверены (доки, issues, исходники)
- Цитаты ключевых строк кода или данных

### 3.3 [Выводы]
- Почему выбран именно этот подход
- Минимум 2 отклонённые альтернативы с конкретными причинами отказа
- Ссылки на подтверждающие данные из раздела Исследование

### 3.4 [Реализация]
- КАЖДОЕ изменение с `файл:строка`
- Что делает каждая строка/блок кода
- Какие импорты добавлены и зачем
- Как новый код взаимодействует с существующим
- Примеры использования (входные/выходные данные)

### 3.5 [Проверка]
- Какие тесты запущены (команда, результат)
- Какие сценарии вручную проверены
- Ожидаемый результат vs фактический
- Скриншоты или логи если применимо

### 3.6 [Риски]
- Побочные эффекты на другие модули
- Обратная совместимость (Desktop, CLI, Manager)
- Влияние на производительность (размер БД, скорость запросов)
- Что может сломаться при обновлении opencode

### 3.7 [Ссылки]
- Doc-ID всех связанных документов
- Ссылки на issues, PR, коммиты
- Ссылки на внешнюю документацию

## 4. Architecture Documentation (ОБЯЗАТЕЛЬНО)

При добавлении НОВОГО функционала:

1. **feature doc:** `docs/features/<name>.md` с Doc-ID (минимум 100 строк)
   - Проблема и её контекст
   - Альтернативные решения и почему отклонены
   - Архитектурное решение (диаграмма ascii, схема данных)
   - Сложности (min 3) с описанием как решены
   - Примеры использования (до/после, вход/выход)

2. **reference doc:** `docs/reference/<topic>.md` если затрагиваются:
   - Новые таблицы или колонки в SQLite
   - Новые алгоритмы (project resolution, hash computation)
   - Новые протоколы (MCP, HTTP API)

3. **changelog:** Обнови `docs/CHANGELOG.md` с указанием Doc-ID

4. **help tab:** Обнови справку в `app.py` (раздел 2.x или новый)

5. **SQL queries:** Если затрагивает read-only операции — добавь запросы в `reference/sql-queries.md`

6. **troubleshooting:** Если затрагивает диагностику — добавь сценарий в `guides/troubleshooting.md`

## 5. Documentation Quality Gate

Перед завершением таска ОБЯЗАТЕЛЬНО проверь:

- [ ] Создан/обновлён feature-документ в `docs/features/` (≥100 строк)
- [ ] Обновлён `docs/CHANGELOG.md`
- [ ] Обновлена справка в `app.py`
- [ ] Если новый тип данных — создан/обновлён `docs/reference/`
- [ ] Если новая проблема — создан/обновлён `docs/guides/troubleshooting.md`
- [ ] Doc-ID указаны в заголовке документа и в списке связанных
- [ ] Все изменения запушины в git
